4 Reference Implementation

Basketball is the recommended first non-soccer reference implementation. It has a clean team-versus-team event structure, no draw outcome in most competitions, and enough overlap with the World Cup workflow to prove the reusable ingestion, validation, modeling, and reporting layers.

The first production pass for each sport will use the same sequence:

  1. Ingest raw API snapshots with timestamped manifests.
  2. Build cleaned game, team, player, standings, odds, and statistics tables where available.
  3. Create model-ready rows with one row per team-game or event participant.
  4. Run validation tests before modeling.
  5. Fit baseline models first, then Bayesian models where the sport has enough historical depth.
  6. Backtest by time, compare calibration, and publish small result samples.

5 Current Lineage Rules

  • Raw JSON stays local in data/raw/apisports_inventory/.
  • Sanitized summaries are written to data/processed/multisport/.
  • Published pages show samples, endpoint status, timestamps, and field definitions.
  • The public site shows compact summaries only; full raw feeds remain in local project storage.

slug <- function(x) {
  x <- iconv(as.character(x), to = "ASCII//TRANSLIT")

}

escape_html <- function(x) {
  x <- ifelse(is.na(x), "", as.character(x))
  x <- gsub("&", "&amp;", x, fixed = TRUE)
  x <- gsub("<", "&lt;", x, fixed = TRUE)
  x <- gsub(">", "&gt;", x, fixed = TRUE)
  x <- gsub('"', "&quot;", x, fixed = TRUE)
  x
}

## Sport Inventory

## Reference Implementation

Basketball is the recommended first non-soccer reference implementation. It has a clean team-versus-team event structure, no draw outcome in most competitions, and enough overlap with the World Cup workflow to prove the reusable ingestion, validation, modeling, and reporting layers.

The first production pass for each sport will use the same sequence:

1. Ingest raw API snapshots with timestamped manifests.
2. Build cleaned game, team, player, standings, odds, and statistics tables where available.
3. Create model-ready rows with one row per team-game or event participant.
4. Run validation tests before modeling.
5. Fit baseline models first, then Bayesian models where the sport has enough historical depth.
6. Backtest by time, compare calibration, and publish small result samples.

## Current Lineage Rules

- Raw JSON stays local in `data/raw/apisports_inventory/`.
- Sanitized summaries are written to `data/processed/multisport/`.
- Published pages show samples, endpoint status, timestamps, and field definitions.
- The public site shows compact summaries only; full raw feeds remain in local project storage.
