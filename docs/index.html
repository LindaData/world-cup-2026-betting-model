This project builds a reproducible data and modeling workflow for the 2026 FIFA World Cup. The current public version is a transparent baseline, not a final betting system.

The project is designed around four layers:

  • Data collection: public match history, Elo ratings, 2026 venues, weather, player metadata, and news signals.
  • Data storage: local files and DuckDB tables for modeling.
  • Modeling: interpretable first-pass models in R, with Python handling API pulls and heavier data jobs.
  • Publishing: Quarto renders a shareable GitHub Pages site from the local analysis.

The starting modeling unit is one row per team-match. A single game creates two observations, one for each team.

goals_metrics <- read_csv_if_exists(file.path(root, "data", "processed", "modeling", "goals_linear_model_metrics.csv"))

if (nrow(goals_metrics) > 0 && nrow(ordinal_metrics) > 0) {
  data.frame(
    model = c("Goals OLS", "Ordinal result logistic"),
    target = c("team goals", "loss < draw < win"),
    rows_total = c(goals_metrics$rows_total[[1]], ordinal_metrics$rows_total[[1]]),

4 Interactive App Direction

4.1 Shiny First, Streamlit Later If Needed

Your older Shiny code used navbarPage(), sidebarLayout(), DT tables, model controls, and MathJax. That is the right pattern for the local interactive version because you already work in RStudio.

GitHub Pages can host this static Quarto report for free. A true Shiny app needs a running R server, usually ShinyApps.io, Posit Connect, or your own cloud VM. Streamlit would have the same hosting issue, so it is not better for the public static site yet.
